CHAPTER 01

概念映射 — Web 技术 vs C++

作为 C++ 开发者，你已经掌握了编程的核心概念：变量、函数、类、编译、链接、内存管理。 好消息是，Web 开发的每一个概念在 C++ 中几乎都有对应物。 本章建立一张"翻译表"，让你可以用已有的知识框架来理解全新的技术栈。

核心概念对照表

下表是 Web 技术与 C++ 的核心概念映射。浏览一遍即可，后续章节会逐一展开。

一句话总结：Web 开发就是 HTML（声明结构）+ CSS（配置外观）+ JS（编写逻辑），三者协作，由浏览器这个"运行时"统一执行。这就像 C++ 中你定义 struct（数据结构）、配置 iostream 格式（显示方式）、编写 .cpp 逻辑，最终由操作系统执行。

编译型 vs 解释型：核心差异

这是 C++ 与 Web 最根本的区别：

C++ 工作流 
// 1. 编写源码
//    main.cpp, utils.h, utils.cpp

// 2. 预处理 → 编译 → 链接
//    g++ -o app main.cpp utils.cpp

// 3. 生成二进制可执行文件
//    ./app  (机器码，直接运行)

// 改一行代码？→ 重新编译整个项目
Web 工作流 
// 1. 编写源码
//    index.html, layout.css, main.js

// 2. 无编译步骤！
//    浏览器直接读取源码文件

// 3. 浏览器实时解析并渲染
//    打开 index.html 即可看到结果

// 改一行代码？→ 刷新浏览器即可

没有编译步骤是最大的不同。C++ 需要 g++ 编译成机器码，而 Web 源码直接被浏览器解释执行。这意味着：无需等待编译、改动即时可见、但运行速度不如原生。就像 Python 之于 C++，但浏览器的 JS 引擎（V8）经过 JIT 优化后性能相当不错。

关键区别：C++ 项目的 .h 和 .cpp 通过 #include 和链接器组合成一个可执行文件。而 Web 项目中，每个 HTML 页面是独立的"入口"，通过 <link> 引入 CSS、通过 <script> 引入 JS。浏览器加载一个 HTML 文件后，会依次加载它引用的所有 CSS 和 JS——这就像一个"运行时链接器"。

"编译"过程：浏览器如何处理你的代码

当用户在浏览器输入网址访问 index.html 时，浏览器的工作流程如下：

浏览器处理流程（类比 C++ 编译流程） 
// C++ 编译流程：
//   源码 → 预处理 → 编译 → 汇编 → 链接 → 可执行文件 → 运行

// 浏览器处理流程：
//   ① 下载 index.html
//   ② 解析 HTML → 构建 DOM 树（类似 AST）
//   ③ 遇到 <link href="layout.css"> → 下载并解析 CSS → 构建 CSSOM 树
//   ④ DOM + CSSOM → 合成渲染树（Render Tree）→ 计算布局 → 绘制像素
//   ⑤ 遇到 <script src="main.js"> → 下载并执行 JS
//   ⑥ JS 可以操作 DOM（修改树节点）→ 触发重新渲染

// 核心区别：
//   C++：编译一次，运行多次，运行时无法修改代码
//   Web：每次打开页面都重新"编译"，JS 可以随时修改 DOM 触发重渲染
展开：为什么 Web 要分 HTML / CSS / JS 三种语言？

这是 Web 最重要的设计哲学——关注点分离（Separation of Concerns）：

HTML（结构）：描述页面"有什么"。类比 C++ 中的 struct 和数据声明——纯数据结构，不关心显示方式和行为。

CSS（样式）：描述页面"长什么样"。类比 C++ 中的 operator<< 重载或 ncurses 的颜色/布局配置——控制视觉呈现。

JavaScript（行为）：描述页面"做什么"。类比 C++ 的 .cpp 实现文件——所有逻辑、交互、网络请求都在这里。

这种分离在 C++ 中也有体现：你不会把 GUI 样式写在业务逻辑里，就像 Qt 中 .ui（界面）、.qss（样式）、.cpp（逻辑）是分开的。Web 从一开始就将此设计为语言级别的分离。

展开：Web 项目的"CMakeLists.txt"在哪里？

答案：HTML 文件本身就是构建配置文件。

C++ 项目用 CMakeLists.txt 声明源文件、头文件路径、链接库。而 HTML 文件里的 <link> 和 <script> 标签就起到了同样的作用：

html — index.html（相当于 CMakeLists.txt）
<!-- 引入 CSS = add_include_directories() -->
<link rel="stylesheet" href="css/layout-new.css">
<link rel="stylesheet" href="css/responsive-new.css">
<link rel="stylesheet" href="css/animations-new.css">

<!-- 引入 JS = target_link_libraries() -->
<script type="module" src="js/supabase-client.js"></script>
<script type="module" src="js/main.js"></script>

本项目不需要 npm、webpack 等构建工具——纯静态文件，浏览器直接加载。这是最简单的 Web 项目形式，没有"构建步骤"。

CHAPTER 02

HTML — 像写 struct 一样写页面

HTML（HyperText Markup Language）是一种标记语言，不是编程语言。 它不能做循环、不能做判断、没有变量——它只是声明页面中"有什么东西"。

如果你用过 XML、Protocol Buffers 或 JSON Schema，那 HTML 的本质和它们一样——是一种结构化的数据描述格式。就像 .proto 文件定义消息结构，HTML 定义页面结构。

标签 = 嵌套的结构体

HTML 由标签（tag）组成，标签可以嵌套——就像 C++ 中 struct 可以包含其他 struct。

C++ — 结构体嵌套 
struct Logo {
  std::string image_src;
  std::string text;
};

struct MenuItem {
  std::string href;
  std::string label;
};

struct Nav {
  Logo logo;
  std::vector<MenuItem> menu;
};

struct Header {
  Nav nav;
};
HTML — 标签嵌套 
<header class="header">
  <nav class="header__nav">
    <div class="header__logo">
      <img src="logo.png"
           alt="411吉他工作室">
      <a href="index.html">
        411吉他工作室
      </a>
    </div>
    <ul class="header__menu">
      <li><a href="index.html">首页</a></li>
      <li><a href="booking.html">维修预约</a></li>
    </ul>
  </nav>
</header>

属性 = 成员变量：HTML 标签上的 class、href、src 等属性，就像 struct 的成员变量。<img src="logo.png"> 相当于 Image img; img.src = "logo.png";

DOM 树 = 内存中的抽象语法树

浏览器解析 HTML 后，会在内存中构建一棵DOM 树（Document Object Model）。这与 C++ 编译器解析代码生成 AST 完全类似：

DOM 树结构（对应上面的 HTML） 
document                        // 根节点，类似 TranslationUnit*
└── html
    ├── head
    │   └── title → "411吉他工作室"
    └── body
        ├── header .header
        │   └── nav .header__nav
        │       ├── div .header__logo
        │       │   ├── img src="logo.png"
        │       │   └── a → "411吉他工作室"
        │       └── ul .header__menu
        │           ├── li → a → "首页"
        │           └── li → a → "维修预约"
        └── main
            └── ...

每个 DOM 节点就是内存中的一个对象（Node*），有 parentNode（父指针）、childNodes（子节点列表）、attributes（属性 map）。JS 通过 document.querySelector() 获取节点引用，就像 C++ 中通过遍历 AST 获取节点指针。

CHAPTER 03

CSS — 像配置文件一样控制样式

CSS（Cascading Style Sheets）用来控制 HTML 元素的视觉呈现——颜色、大小、布局、动画。 它不包含任何逻辑，只是一组声明式的规则。

如果你用过 Qt，CSS 的角色就像 .qss 样式表文件。如果你用过游戏引擎，CSS 类似于材质/皮肤配置文件。它的本质是：根据选择器匹配目标元素 → 对其应用一组键值对属性。就像 C++ 中 std::map<string, string> style;

CSS 规则的结构

每条 CSS 规则由选择器和声明块组成：

CSS — 基本语法 
/* 选择器 { 属性: 值; } */

.hero__title {
  font-size: 3rem;
  color: #c9a227;
  font-weight: 700;
  margin-bottom: 1.5rem;
}
C++ — 等价操作 
// 找到所有 class="hero__title" 的元素
// 然后给它们设置属性

for (auto& el : querySelectorAll(".hero__title")) {
  el.style["font-size"]   = "3rem";
  el.style["color"]       = "#c9a227";
  el.style["font-weight"] = "700";
  el.style["margin-bottom"] = "1.5rem";
}

CSS 选择器 = 模式匹配

选择器决定"哪些元素"会被应用样式。就像 C++ 模板特化中的模式匹配——越具体的规则优先级越高。

选择器C++ 类比匹配规则示例
div 按类型匹配所有 匹配所有该标签的元素 div { ... } 匹配所有 <div>
.hero find_if(class=="hero") 匹配 class 包含 hero 的元素 .hero { ... } 匹配 <div class="hero">
#calendar-modal 唯一标识符查找 匹配 id 为指定值的唯一元素 #calendar-modal { ... }
.header .logo 树上的路径查找 匹配 .header 内部的 .logo 后代选择器（任意深度嵌套）
.btn:hover 条件触发 鼠标悬浮时应用 伪类选择器（状态匹配）

优先级（Specificity）类似 C++ 模板特化规则：越具体的声明越优先。#id（最具体）> .class > tag（最泛化）。如果两条规则优先级相同，后写的覆盖先写的——类似链接器中后出现的符号覆盖先出现的。

CSS 变量 = 全局常量 / 配置宏

本项目大量使用 CSS 变量（Custom Properties），定义在 :root（根元素）上。这就像 C++ 项目中的全局 constexpr 常量或 #define 宏：

CSS — 本项目 layout-new.css 
/* 全局设计令牌（Design Tokens）*/
:root {
  /* 颜色 - 温暖木质琥珀色主题 */
  --color-bg-primary: #0f0d0a;
  --color-bg-secondary: #1a1714;
  --color-text-primary: #f8f5f0;
  --color-accent-primary: #c9a227;

  /* 字体 */
  --font-display: 'Cormorant Garamond', serif;
  --font-body: 'Noto Sans SC', sans-serif;

  /* 间距 */
  --spacing-md: 1rem;
  --spacing-lg: 1.5rem;

  /* 圆角 */
  --radius-md: 0.375rem;
}

/* 使用变量 */
.header {
  background: var(--color-bg-secondary);
  color: var(--color-text-primary);
  padding: var(--spacing-lg);
}
C++ — 等价的常量配置 
// config.h — 全局设计令牌
namespace theme {
  constexpr auto bg_primary  = "#0f0d0a";
  constexpr auto bg_secondary = "#1a1714";
  constexpr auto text_primary = "#f8f5f0";
  constexpr auto accent       = "#c9a227";

  constexpr auto font_display = "Cormorant";
  constexpr auto font_body    = "Noto Sans";

  constexpr float spacing_md = 16.0f;
  constexpr float spacing_lg = 24.0f;

  constexpr float radius_md = 6.0f;
}

// 使用常量
header.bg      = theme::bg_secondary;
header.color   = theme::text_primary;
header.padding = theme::spacing_lg;

集中管理的优势：想把整个网站从金色主题改为蓝色？只需修改 :root 里的几个变量值——不需要全局搜索替换。这和 C++ 中把 #define COLOR_PRIMARY 改成新值，然后重编译的效果一样。但 CSS 变量更强大——它们可以运行时修改（通过 JS 改 style.setProperty），这在 C++ 中是 constexpr 做不到的。

BEM 命名法 = 命名空间规范

本项目的 CSS 类名遵循 BEM（Block__Element--Modifier）命名规范：

BEM 命名对照 
/* Block = 独立组件（class）*/
.header  { ... }            /* ←→ class Header { }; */

/* Element = 组件内部子元素（成员）*/
.header__nav  { ... }       /* ←→ Header::nav */
.header__logo { ... }       /* ←→ Header::logo */
.header__menu { ... }       /* ←→ Header::menu */

/* Modifier = 变体/状态（模板特化）*/
.btn--primary  { ... }      /* ←→ Button<Primary> */
.btn--disabled { ... }      /* ←→ Button<Disabled> */
.card--featured { ... }     /* ←→ Card<Featured> */

BEM 解决的问题就像 C++ 中的命名空间：避免命名冲突。没有 BEM 时，.title 可能在页头和卡片中冲突；有了 BEM，.header__title 和 .card__title 清晰隔离，就像 Header::title 和 Card::title。

JavaScript 基础 — 从 C++ 到 JS

JavaScript 是 Web 的唯一编程语言。如果说 HTML 是 struct 声明、CSS 是样式配置， 那么 JS 就是你的 .cpp 实现文件——所有逻辑、交互、网络请求都在这里完成。

IntersectionObserver = 滚动监控器

本项目使用 IntersectionObserver 实现滚动动画——当元素进入视口时触发回调：

JavaScript — main.js 滚动动画 
function initScrollReveal() {
  // 1. 查找所有带 .scroll-reveal 类的元素
  const elements = document.querySelectorAll('.scroll-reveal');

  // 2. 创建观察器（类似 C++ 中注册一个定时回调）
  const observer = new IntersectionObserver(
    (entries) => {
      entries.forEach((entry) => {
        if (entry.isIntersecting) {
          // 3. 元素进入视口 → 添加 CSS 类触发动画
          entry.target.classList.add('is-visible');
          observer.unobserve(entry.target); // 只触发一次
        }
      });
    },
    { threshold: 0.1 }  // 元素 10% 可见时触发
  );

  // 4. 对每个元素启动观察
  elements.forEach((el) => observer.observe(el));
}

// 工作流程：
// HTML: <div class="scroll-reveal">内容</div>
// CSS:  .scroll-reveal { opacity: 0; transition: opacity 0.6s; }
//       .scroll-reveal.is-visible { opacity: 1; }
// JS:   滚动到元素位置 → 添加 is-visible → CSS 动画自动播放

职责分离：注意 JS 只负责"在正确的时机添加 CSS 类"，动画本身由 CSS 的 transition 属性完成。这是 Web 开发的常见模式——JS 控制状态（添加/移除类），CSS 控制外观（动画/样式）。类比 C++ 中的 MVC 模式：JS 是 Controller，CSS 是 View，HTML 是 Model。

展开：DOM 性能注意事项

频繁修改 DOM 会导致浏览器不断重新计算布局和重绘——这和 C++ 中频繁 malloc/free 类似，会产生性能问题。优化策略：

1. 批量修改：先构建完整 HTML 字符串，再一次性赋值 innerHTML（本项目就是这样做的）。不要在循环中逐个 appendChild。

2. DocumentFragment：如果必须逐个添加节点，用 DocumentFragment 作为临时容器（类似 C++ 中先构建子树再挂载）。

3. 虚拟列表：大量数据（如 1000+ 条订单）不要全部渲染，只渲染可见区域。

4. requestAnimationFrame：动画相关的 DOM 修改放在 requestAnimationFrame 回调中，确保与屏幕刷新率同步（类似游戏引擎的渲染循环）。

类比 C++：事件冒泡就像异常传播——如果当前函数不处理，就向上抛给调用者。事件委托则像在基类中统一处理消息分发。

前后端通信 — Supabase 数据库

到目前为止，我们讲的都是前端（浏览器中运行的 HTML/CSS/JS）。 但一个真正的应用需要持久化数据——用户提交的预约订单、管理员的操作记录。 这就需要后端——服务器上运行的数据库和 API。

对 C++ 开发者来说，"后端"就是一个远程进程，通过网络暴露 API。你的前端代码（类似 C++ 客户端程序）通过 HTTP 请求与之通信，就像 C++ 中通过 Socket / gRPC / REST 调用远程服务。唯一的区别是：C++ 中你可能直接链接 libpq 操作 PostgreSQL，而 Web 前端不能直连数据库——必须通过 HTTP API 中转。

本项目使用 Supabase——一个 BaaS（Backend as a Service）平台。它在云端为你提供：

RLS = 数据库级别的权限控制

Row Level Security 是 PostgreSQL 的特性——在数据库层面控制每行数据的访问权限：

SQL — 本项目的 RLS 策略 
-- 启用行级安全
ALTER TABLE guitar_repairs ENABLE ROW LEVEL SECURITY;

-- 策略 1：任何人都可以创建订单（游客也行）
CREATE POLICY "Anyone can create"
ON guitar_repairs FOR INSERT
WITH CHECK (true);           -- 无条件允许

-- 策略 2：匿名用户也可以查看（客户查单）
CREATE POLICY "Anon can view"
ON guitar_repairs FOR SELECT
USING (auth.role() = 'anon');

-- 策略 3：已认证用户可以查看/更新所有订单（管理员）
CREATE POLICY "Auth can manage"
ON guitar_repairs FOR ALL
TO authenticated
USING (true);

/* C++ 等价（中间件伪代码）：
   Result query(User user, string sql) {
     if (sql.isInsert()) return db.exec(sql);     // 任何人可 INSERT
     if (user.role == ANON)  // 匿名用户只能 SELECT
       return db.exec(sql + " WHERE customer_phone = ?", user.phone);
     if (user.role == AUTHENTICATED) return db.exec(sql); // 管理员无限制
   }
*/

安全边界：在传统 C++ 后端中，你需要在应用层写权限检查代码。Supabase 的 RLS 把这个逻辑下沉到了数据库层——即使有人绕过前端直接调用 API，数据库也会拒绝未授权的操作。这就像把 assert 放在数据层而不是业务层——更安全。

展开：为什么浏览器不能直连数据库？

C++ 程序可以直接通过 TCP Socket 连接 PostgreSQL（libpq），但浏览器中的 JS 不能——这是出于安全考虑：

1. 暴露凭据：浏览器的所有代码对用户完全透明（F12 就能看到）。如果在 JS 中放数据库密码，任何人都能拿到。

2. 无限权限：直连数据库意味着用户可以执行任意 SQL（DROP TABLE!）。

3. 浏览器限制：浏览器只允许 HTTP/HTTPS/WebSocket 协议，不支持原始 TCP 连接。

所以需要一个中间层（API 服务器 / Supabase）——它验证身份、检查权限、过滤危险操作，然后代替前端与数据库通信。Supabase 的 anon key 不是数据库密码——它是一个权限受限的 JWT 令牌，只能执行 RLS 策略允许的操作。

展开：Supabase 的 anon key 安全吗？

你可能注意到 supabase-client.js 中硬编码了 SUPABASE_ANON_KEY。这个 key 暴露在前端代码中——但这是设计如此：

anon key 是一个受限的 JWT 令牌，只拥有 anon 角色的权限。它能做什么完全由 RLS 策略决定。

本项目中，anon 角色可以：创建维修订单（INSERT）、查看订单（SELECT）。不能：修改订单状态（UPDATE 需要 authenticated 角色）、删除订单。

这就像 C++ 中的只读数据库用户——即使密码泄露，也只能执行 SELECT 和限定范围的 INSERT。真正的管理操作需要管理员登录后获得 authenticated 角色的 JWT。

技术决策选择替代方案选择原因

总体理念：本项目追求最小复杂度——没有构建工具、没有框架、没有后端服务器。所有文件可以直接在浏览器中运行。这和 C++ 中"单个 .cpp 文件 + g++ main.cpp -o app"的极简风格类似——当项目足够小时，不需要 CMake、不需要包管理器。

恭喜你完成了本教程！如果想深入 Web 开发，以下是推荐的学习路径：

学习路线图 
你现在的位置（本教程覆盖）
  ✓ HTML 基础
  ✓ CSS 基础 + 变量 + 响应式
  ✓ JavaScript 基础 + async/await
  ✓ DOM 操作 + 事件处理
  ✓ Supabase（BaaS）前后端通信
  ✓ 完整项目架构理解

进阶方向（按需选择）
  → TypeScript         给 JS 添加静态类型（C++ 开发者会很喜欢）
  → React / Vue        组件化框架（适合复杂 UI）
  → Node.js            用 JS 写后端（如果需要自定义 API）
  → Tailwind CSS      实用类优先的 CSS 框架（加速样式开发）
  → Next.js / Nuxt    全栈框架（SSR + 路由 + API）
  → WebAssembly        在浏览器中运行 C++ 代码！你的 C++ 技能直接复用

特别推荐 TypeScript：作为 C++ 开发者，你一定怀念静态类型。TypeScript 就是"带类型的 JavaScript"——它的类型系统在很多方面比 C++ 更强大（联合类型、类型推导、泛型约束）。如果继续做 Web 开发，强烈建议从 TypeScript 开始。

特别推荐 WebAssembly（WASM）：这项技术让你可以把 C++ 代码编译成浏览器能运行的格式——也就是说，你可以用 C++ 写高性能的 Web 应用核心逻辑，然后用 JS/HTML 做 UI。这是 C++ 开发者进入 Web 世界的最自然路径。
